Cetec ERP · MySQL Database

database guide

the tables you'll actually be working with, what they mean, and what to watch out for. this is a living document — part of your job this summer is to fill in what's missing.

heads up: Cetec's database has a lot of columns with misleading names, cached values, and nulls. don't trust a column just because it exists. when in doubt, ask Aliya.
inventory & parts

These tables track parts, their quantities on hand, and bin locations. The most common source of confusion is qoh_cache vs c_fifoix — read the callout below carefully.

critical: always use c_fifoix summed bin quantities for inventory — NOT qoh_cache. qoh_cache is a cached value that lags behind real data and will give you wrong numbers. this has burned us before.
TableWhat It IsKey ColumnsNotes
partmaster Master list of every part in the system partno, description, prc core table — always your starting point for part lookups
c_fifoix FIFO inventory index — live bin quantities partno, qty, bin use this — SUM(qty) grouped by partno is your reliable QOH
partmaster.qoh_cache Cached quantity on hand on partmaster qoh_cache avoid — lags behind real inventory, often stale
part_pick_layers Tracks which FIFO layers were picked for work orders partno, wo_id, qty_picked key join path for linking WOs to PO receipts via FIFO
purchasing & POs

Purchase orders, line items, and receipts. Vendor 712 is excluded from most procurement reports — don't ask why, just always add that filter.

always exclude vendor 712 from any procurement or spend analysis. add WHERE vendor_id != 712 to any PO query.
TableWhat It IsKey ColumnsNotes
po Purchase order header id, vendor_id, po_date, status core table — one row per PO
po_line Line items on each PO po_id, partno, qty_ordered, unit_price join to po on po_id
po_line_receipts Records when PO lines were received into inventory po_line_id, qty_received, receipt_date key for on-time delivery analysis and WO-to-PO linking via FIFO path
vendor Vendor master list id, name, lead_time join on vendor_id — always exclude id=712
work orders & production

Work orders drive production. Linking a work order all the way back to the POs that supplied its materials is one of the hardest join paths in this database — Cetec hasn't given us a clean answer on it yet.

known blocker: there's no direct FK from work orders to POs. the best join path we've found goes through part_pick_layers → po_line_receipts via the FIFO inventory trail. this is still being worked out — documenting what you find is part of your project.
TableWhat It IsKey ColumnsNotes
wo Work order header id, partno, status, open_date, due_date core table — one row per work order
wo_line BOM components needed for each work order wo_id, partno, qty_required what materials are needed to build the WO
wo_labor Labor entries logged against work orders wo_id, employee_id, hours, labor_date needed for OLE analysis — but Paylocity data not yet integrated
part_pick_layers Tracks which inventory layers were consumed by a WO wo_id, partno, fifo_layer_id the bridge table for WO → PO tracing via FIFO path
